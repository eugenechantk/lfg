### 2026-06-29 — flowdeck run wipes sim prefs on code-change reinstall
**What happened:** To verify the directory-grouping UI against the live host, I set `lfg.baseURL` via PlistBuddy in the session sim's data container, but every `flowdeck run` (which rebuilds+reinstalls on a code change) wiped UserDefaults, dropping back to the "Connect to your host" screen. Burned ~4 cycles before driving the connect text field directly with `flowdeck ui simulator type` (which the project memory warns is unreliable for URLs, but worked fine for a loopback URL).
**Why this was slow:** Assumed the plist trick (project CLAUDE.md) would survive reinstall like it did once early on; it doesn't survive a fresh install that replaces the container.
**What better looks like:** When the app needs a configured host for visual verification AND the build is changing between captures, set the host once, then do all UI verification without another `flowdeck run`. If a rebuild is unavoidable, just type the loopback URL into the connect field — it's reliable for short non-tailnet URLs. Consider a debug launch-arg/env to seed the host so verification never needs the connect flow.

### 2026-06-29 — TestFlight setup hit a chain of predictable, individually-fixable failures
**What happened:** Getting LFG's first build to TestFlight required six deploy re-runs, each surfacing the next blocker: (1) App Store Connect API can't CREATE app records (Apple limitation), (2) App ID lacked Push capability so the wildcard profile failed, (3) automatic-signing distribution export can't resolve a profile headlessly, (4) `get_provisioning_profile` rejects `development:`+`adhoc:` passed together, (5) AppIcon.appiconset had a Contents.json but no PNG → 120x120 validation failure.
**Why this was slow:** Each was discovered serially by running the full pipeline to the next break. Several were knowable up front by inspecting state before the first archive: the empty appiconset (no .png file), the App ID's capability set, and the entitlement requiring aps-environment.
**What better looks like:** Before the first `deploy_testflight`, run a preflight: (a) assert AppIcon.appiconset contains an actual image referenced by filename; (b) if the entitlement requests `aps-environment`, ensure the App ID has Push enabled; (c) for local distribution prefer `sigh`+manual signing from the start (automatic-signing export is unreliable headless); (d) know Apple's API cannot create app records — create it (UI/produce) as step 0. Candidate: add a "TestFlight preflight" checklist to the fastlane-match-ci skill.

- [x] 2026-06-29 — Bundle-id change silently broke APNs (LFG_APNS_TOPIC not updated)

### 2026-06-29 — Changing the iOS bundle id silently broke push (stale LFG_APNS_TOPIC)
**What happened:** After changing the bundle id dev.omg.lfg -> com.eugenechan.lfg for TestFlight, notifications stopped on the TestFlight build. Root cause: `.env` `LFG_APNS_TOPIC=dev.omg.lfg` (old id). APNs requires `apns-topic` == the token's bundle id, so every push 400'd (DeviceTokenNotForTopic). The env reporting (#if DEBUG sandbox/else production) and host selection were all correct.
**Why it was easy to miss:** The bundle-id change is in project.yml; the dependent APNs topic lives in server `.env`. Nothing links them, and the apns.ts default fallback is also the old id.
**What better looks like:** Treat LFG_APNS_TOPIC as a derived value of the app bundle id — whenever the bundle id changes, update `.env` topic in the same change. Verified the fix by sending a real APNs push via src/push/apns.ts to the registered production token (200 OK) rather than trusting the config edit — real-seam verification. Note: the watcher caches apnsConfigFromEnv() at startup, so `.env` changes need a `lfg serve` restart to take effect live.

### 2026-06-29 — UI automation: tap menu rows by exact label, not by point
**What happened:** Verifying the Debug copy rows, I tapped by --point coordinates and missed repeatedly (even opened the Rename dialog by accident), because SwiftUI Menu row positions don't map cleanly to guessed points. Tapping `flowdeck ui simulator tap "<exact label>"` worked first try — but the Claude-id row's label was the *full* untruncated id, which I only got once I recalled the session's full id.
**What better looks like:** For menu/list rows, tap by the exact accessibility label (capture the a11y tree from `screen` to read full labels, since the screenshot truncates them) rather than guessing coordinates. Only fall back to --point for icon-only controls with no label (e.g. the ellipsis button).
