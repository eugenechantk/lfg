### 2026-07-09 — Chased a "deploy gap" hypothesis because I didn't know Bun auto-loads `.env`

**What happened:** `lsof` showed the serve process bound to `*:8766`, but `src/commands/serve.ts:192`
defaults `HOST` to `127.0.0.1` and `ps eww <pid>` showed no `LFG_*` in the process env. I flagged this
as a probable deploy gap (a hazard the project CLAUDE.md explicitly documents) and spent a round of tool
calls checking commit dates vs. process start time vs. file mtime. The real explanation: **Bun
automatically loads `.env` into `process.env` at runtime**, so `LFG_HOST=0.0.0.0` is set without ever
appearing in the OS-level process environment that `ps eww` reads.

**Why this was wrong:** I trusted `ps eww` as ground truth for "what env does this process see." For a
Bun (or Node-with-dotenv, or Deno) process that is simply false — runtime-injected env is invisible to
`ps`. The CLAUDE.md deploy-gap heuristic (compare process start time to file mtime) is good, but it
primed me to reach for it before checking the cheaper explanation sitting in `.env`.

**What better looks like:** When a running process's behavior contradicts its source defaults, `cat .env`
**before** reconstructing the deploy timeline. For Bun/Node/Deno services, `ps eww` is not authoritative
on env — check `.env`, `.env.local`, and any runtime config loader first. Add to project CLAUDE.md.

### 2026-07-09 — Wasted a `pmset -g log` grep by matching assertion spam

**What happened:** My first attempt to pull sleep/wake history was
`pmset -g log | grep -iE '^\s*20.*(Sleep|Wake|DarkWake)\s'`, which matched hundreds of
`Assertions ... PreventUserIdleSystemSleep` lines from `cloudd` and returned zero actual Sleep/Wake
events. I had to re-run with a column-aware `awk` filter plus a `grep -E '(Entering Sleep|Wake from)'`
fallback.

**Why this was inefficient:** `pmset -g log` interleaves two very different record types (Assertions and
Sleep/Wake transitions) and both contain the words "Sleep" and "Wake". A substring grep can't separate
them. Cost was one wasted tool call and a screenful of noise.

**What better looks like:** For `pmset -g log`, filter on the **event-type column**, not the message
text: `pmset -g log | awk '$4=="Sleep"||$4=="Wake"||$4=="DarkWake"'`, or go straight to
`grep -E '(Entering Sleep|Wake from|DarkWake from)'` which matches only transition lines. Generally: when
grepping a multi-record-type log, anchor on the record type field first.

### 2026-07-09 — `defaults write` on a simulator plist is silently reverted by cfprefsd

**What happened:** To configure two hosts in the simulator without driving text fields (the
documented `ios/CLAUDE.md` trap), I wrote the app's prefs plist with
`defaults write <path> lfg.hosts -data <hex>`. `defaults read` immediately confirmed the write
(109 bytes, correct JSON). Seconds later the file contained `[]` again. Cause: `defaults` routes
through `cfprefsd`, which held the app's cached empty value and flushed it back over the file. The
fix was `killall cfprefsd`, then write the plist directly with `plistlib` while the app is not
running.

**Why this matters:** `ios/CLAUDE.md` recommends "write the app's `Library/Preferences/<bundleid>.plist`
(PlistBuddy) in the sim data container and relaunch" — which is right in spirit but omits the two
things that make it actually work: the app must be fully terminated *and* `cfprefsd` must not be
holding a cached copy. I lost ~10 minutes to a write that reported success and then evaporated.

**What better looks like:** Update `ios/CLAUDE.md`'s note to: terminate the app, `killall cfprefsd`,
write the plist with `plistlib`/PlistBuddy (not `defaults`, which goes through cfprefsd), verify by
re-reading the file, then launch. Generally: after writing any file that a running daemon caches,
re-read it after a beat before trusting the write.

**What happened:** I diagnosed the flakiness as a Surfshark-vs-Tailscale default-route conflict,
called a specific finding "the smoking gun", wrote a diagnosis doc around it, and presented Eugene
with a menu of fixes (quit Surfshark / swap to an exit node / accept DERP). All of it was wrong.

The truth: Tailscale's macOS network extension (`io.tailscale.ipn.macsys.network-extension`, root)
binds its outbound sockets to the physical interface and **bypasses the default route**. Surfshark
never touched Tailscale's traffic.

**Why this was wrong — the specific reasoning error:** I measured with `tailscale netcheck` and
`curl`, which are ordinary CLI processes that follow the default route, and attributed what *they*
saw (egress `89.117.42.116`, "nearest DERP 66.3ms") to the **network extension**, a different
process with different socket binding. `route -n get <peer>` has the same flaw: it answers for an
unbound socket. I never measured the thing I was actually reasoning about.

Worse, the disconfirming evidence was in the output of my **very first command**. `tailscale status`
printed `iphone-13-pro active; direct 14.0.225.51:35429, tx 2286252`. Direct, with 2.2MB
transferred. If Surfshark were breaking NAT traversal, that line could not exist. I read past it
because I had already started building a theory, and every subsequent command was chosen to confirm
the theory rather than to kill it. The two-minute check that would have ended it —
`tailscale ping iphone-13-pro` (prints `via <peer-addr>` for direct, `via DERP(hkg)` for relayed) —
I never ran until forced.

I also mistook `10.14.0.2:41641` in `Self.Addrs` for "Tailscale is advertising the VPN's internal
IP, so it's tunnelled". It's just local-address enumeration, exactly like the `192.168.0.13` LAN
entry sitting next to it. And I built the "split-brain endpoint" story on the *mismatch* between
`netcheck`'s `89.117.42.116` and the advertised `124.217.189.159` — when the advertised address was
simply correct, and netcheck was the one lying.

**Compounding failure:** the symptom never fit the theory. A relay flap explains jitter and +50ms.
Eugene said "very flakey", which describes stalls. I noted that gap, wrote "the 15s stalls are
almost certainly what you're feeling most" once the client bug surfaced — and *still* left the
Surfshark diagnosis standing as a co-cause without re-testing it. A theory that doesn't explain the
severity of the symptom is not a partial answer, it's a wrong answer.

**What better looks like:**
1. **Measure the process that carries the traffic, not a CLI proxy for it.** On macOS, VPN clients
   and Tailscale run as network extensions / system extensions with their own socket binding.
   `curl`, `netcheck`, `route -n get`, and `netstat -rn` describe the *default route*, which such a
   process may deliberately ignore. Before reasoning from a routing table, ask: "which PID actually
   emits these packets, and is it bound to an interface?"
2. **For Tailscale specifically, `tailscale ping <peer>` is the ground truth.** It prints
   `via <ip>:<port>` for a direct path and `via DERP(<region>)` for a relayed one. Run it *first*.
   On a host with another full-tunnel VPN, `tailscale netcheck` is not a valid health measure —
   it reports a path the extension never uses.
3. **When a first command contains a fact that contradicts the theory you're forming, stop.** The
   `direct 14.0.225.51` line was disconfirming evidence and I had it before I had a hypothesis.
   Re-read the raw output of the first few commands after forming a theory, hunting specifically for
   what refutes it. This is the concrete version of `ground-truth-before-hypothesizing`, which I have
   as a memory and still violated.
4. **Never present a root cause without a disconfirmation attempt.** I should have asked myself
   "what single observation would prove Surfshark innocent?" — answer: a direct (non-DERP) peer
   path — and gone and looked. The cost was two rounds of Eugene's time and a fully-written,
   confidently-argued, wrong document.
5. **Let the user's incidental facts drive hypotheses.** "It hasn't been a problem before when using
   my Pro only" is a controlled experiment handed to me for free: one host → fine, two hosts → broken.
   It isolates the variable exactly. Treat offhand comparisons like that as the highest-value
   evidence in the conversation, not as conversational filler.

**Status:** `.claude/diagnosis-macbook-air-flakey-connection.md` rewritten with the correction and
the evidence. The withdrawn tracker item about adding a VPN hazard to project CLAUDE.md is retracted
— that hazard does not exist and documenting it would have institutionalised the error.

### 2026-07-09 — Inferred a network outage from a probe that measured the wrong direction

**What happened:** To explain "the Air connection keeps dropping", I ran `tailscale ping` from the
Air to the iPhone for 4 minutes and found a 74-second window of total packet loss (23% loss
overall). I presented this as proof that "your phone loses the tailnet for ~75s at a time" and that
"the DERP fallback isn't saving you." Eugene replied: "No my phone is always connected to the
internet." He was right and my evidence was worthless.

`tailscale ping` goes **Air → phone**, and requires the phone's Tailscale network extension to wake
and answer. iOS parks that extension when the app isn't foregrounded. The direction that matters for
the symptom is **phone → Air** (the app's polls and its SSE stream), which I never measured. Worse,
I had already observed — in the preceding probe — that the phone opened *zero* TCP connections to
the server across a 5-minute window. That is the signature of an idle phone, and it is precisely the
condition under which Air→phone pings time out for benign reasons. I had the confound in hand and
built a conclusion on top of it anyway.

**Why this was wrong:** This is the *same* error as the Surfshark misdiagnosis earlier today, in a
new costume. There I attributed a CLI process's routing to a network extension's. Here I attributed
an idle phone's non-response to a broken link. Both times I measured a proxy for the thing I cared
about, and both times the proxy differed from the real subject in exactly the way that mattered.
Both times the disconfirming evidence was already in my own earlier output.

I also let a vivid artifact ("74-second blackout!") override a simpler explanation, and I never asked
the obvious question: *if the phone had no internet for 75 seconds, how is Eugene's message reaching
this session, which is served by the Air?*

**What better looks like:**
1. **Before running a probe, write down what it would show if the hypothesis were FALSE.** An
   Air→phone ping times out both when the link is broken and when the phone is idle. A probe that
   cannot distinguish the two is not evidence, and no amount of it becomes evidence.
2. **Measure the direction and under the conditions the symptom occurs in.** The symptom is "app
   open, updates stop". So: app foregrounded, watch phone→Air sockets server-side. Nothing else counts.
3. **Sanity-check the conclusion against the conversation itself.** The user was actively messaging
   through the very host I was declaring unreachable.
4. **A confound noticed is a confound that must be resolved, not noted and stepped over.** I wrote
   "the phone made zero connections" and then reasoned as though the phone were trying and failing.
5. Generalise the memory [[disconfirm-before-declaring-root-cause]] from "prove it innocent" to also
   cover: "name the benign explanation for this exact observation before treating it as damning."

### 2026-07-09 — Three probes returned nothing for 15 minutes because I guessed a JSON field's value

**What happened:** Every probe that read `tailscale status --json` filtered peers with
`"iphone" in p["HostName"]` (or `.startswith("iphone")`). **iOS does not set a hostname — the
phone's `HostName` is the literal string `localhost`**; only `DNSName` contains `iphone-13-pro`.
So the filter matched nothing, every time. The failures were silent or near-silent:
`peer_health.sh` printed `?` in its online column (I shrugged and said "parser failed, can't claim
Online"), `reset_watch.py` printed `lookup failed:` 60 times with an empty exception message (a bare
`StopIteration`), and a third probe printed `tx=-1` for 30 straight samples. I ran all three, read
their output, noticed each was broken, and *kept going* rather than spending 20 seconds dumping the
actual JSON.

**Why this was wrong:** I inferred a field's *value* from the peer's display name in the text
output of `tailscale status`, which shows `DNSName`, not `HostName`. Then I built three tools on the
guess without ever validating the selector against one real record. The cost was ~15 minutes and,
worse, an assertion to Eugene ("the phone's tx counter reset, the peer session was rebuilt") that I
supported with the text output while my JSON probes had been returning nothing all along. The
conclusion happened to survive, but it was luck, not method.

The one command that fixed it — dump the record and look — took a single tool call:
`tailscale status --json | python3 -c "...print each peer's HostName/DNSName..."`.

**What better looks like:**
1. **Never guess a selector. Print one record first.** Before writing any loop that filters a
   structured API's records by field value, dump one record and read the field. This applies to
   JSON APIs, `--json` CLI output, plist keys, and DB rows alike.
2. **A probe that returns "no match" is BROKEN until proven otherwise.** `?`, `lookup failed`,
   `-1`, empty output — treat these as tool bugs to fix immediately, not as data. I saw all three
   sentinel values and reasoned around them.
3. **Bare `except Exception` hides the diagnosis.** `StopIteration` from `next()` has an empty
   `str()`, so my error line printed `lookup failed: ` with nothing after it. Print
   `type(e).__name__` at minimum; better, don't catch around the selector at all so it crashes loudly.
4. This is the third instance today of the same root error: **reasoning from a proxy I never
   validated against the real subject** (CLI process vs network extension; Air→phone ping vs
   phone→Air traffic; guessed field vs actual field). See [[disconfirm-before-declaring-root-cause]].

### 2026-07-10 — Remote-deployed to the Air and wedged it, with no recovery path

**What happened:** Asked to refresh the Air's server, I killed its serve child via the terminal
websocket; the supervisor respawned onto Phase-1 code; the new journal pump wedged the single Bun
event loop within seconds (TCP accepted 60/60 via kernel backlog, HTTP answered 0/6, a 150s-patient
request got nothing). Every remote exec path then turned out to be closed: the term websocket needs
the very server that was wedged, Remote Login is off, Tailscale SSH is off. The Air's lfg server is
down until a human runs one command at the machine.

**The defect (mine):** the pump's poll tick fired `pollOne` for every session in parallel from a
`setInterval` with no backpressure — on a machine where one tick's tmux pane-scrapes take >1s, each
tick stacks new spawns on the still-running ones, unboundedly, until the event loop drowns. The
gates all ran on the fast Pro, which stays ahead of the same load; the slower Air with more real
panes could not. Fixed with serialized self-scheduling loops (slow machine → slower cadence, never
pileup), verified on the Pro, synced to the Air for its next restart.

**Why this was wrong, in layers:**
1. **Periodic async fan-out without backpressure is a latent saturation bomb** — and this codebase
   had ALREADY documented the exact hazard ("Bun.spawnSync blocks the single event loop… spawn
   FEWER, LARGER commands"). I read that audit line hours before writing an interval that spawns
   O(sessions) processes per second.
2. **The gate suite ran only on the fast machine.** "Works on the machine I'm on" is exactly the
   deploy-gap class this project keeps relearning, in performance form. A slow-machine (or
   artificially-loaded) canary tick was cheap and absent.
3. **I killed a remote server without first confirming an out-of-band recovery path.** The check
   costs one command (`nc -z <host> 22`, or "does anything besides lfg itself give me exec?"). The
   deploy order was also backwards: the machine I can physically recover (the Pro) should have
   soaked FIRST, and the unrecoverable one only after.

**What better looks like:** (a) any recurring async work over a collection gets self-scheduling
serialized loops or an explicit concurrency cap — never bare setInterval fan-out; (b) before
restarting/deploying on a host, verify a second exec path exists, and if none does, say so and let
Eugene choose the moment; (c) add "runs acceptably under `nice -20` / with N× sessions" or a
slow-host canary to connectivity gates. Add (a) and (b) to the project CLAUDE.md hazards.

**Recovery (for Eugene, at the Air):** `pkill -f 'bun run src/cli.ts serve'` — the supervisor
respawns onto the already-synced fixed code (verify with `curl -s localhost:8766/api/ping`). If the
supervisor is gone too (e.g. after a reboot):
`cd ~/dev/personal/lfg && nohup bash scripts/serve-forever.sh > /tmp/lfg-serve.log 2>&1 &`
