### 2026-06-29 — zsh `=name` expansion faked a session-death bug

**What happened:** Ran `tmux has-session -t =lfg-xxx` / `kill-session -t =lfg-xxx` directly in zsh to probe whether idle managed sessions die. zsh EQUALS expansion turns `=word` into "resolve `word` as a command path", erroring `(eval):1: lfg-xxx not found`, which I misread as "tmux session gone → session died while idle."
**Why this was wrong:** The product passes `"=name"` via `Bun.spawnSync([...])` (no shell), so it's immune. Ground truth (`ps -p <pid>`) showed the claude pid alive the whole time and the send `delivered`.
**What better looks like:** When running tmux `-t =name` in zsh, quote it: `-t "=name"`. Cross-check a "process died" claim with `ps -p <pid>` before theorizing.

### 2026-06-29 — Bug fixed: send to a reaped idle session

**Root cause:** When a managed claude session's tmux pane is reaped while idle (box reboot, host restart, manual close), the claude process is gone so it leaves the pgrep-based live list. Two failures stacked: (1) server `/api/sessions/:id/send` returned 404 "session not found"; (2) the iOS app's `refresh()` dropped the session from `store.sessions`, so the detail view collapsed to "No session selected" and the composer disappeared — the user couldn't even attempt a send.
**Fix:**
- Server (`src/commands/serve.ts` + `src/sessions.ts`): factored the resume logic into `resumeClosedSession()` (shared with `/resume`), preserving the session's last model (new `modelAliasForTranscript`). `/send` now, when the session isn't live, resumes the claude transcript with the message as the kickoff prompt and returns `resumed:true` + the (same) sessionId. Verified: HTTP 200, message delivered, claude replies, full conversation continuity, model preserved.
- iOS (`SessionStore`, `LFGClient`, `Models`): `sendMessage` decodes the response and `applyResume` remaps if a new id ever comes back; `focusedSnapshot` keeps the focused session's detail view + composer alive after it leaves the live list so a send can revive it.
**Verification:** End-to-end in the iOS simulator — opened an idle session, killed its tmux pane (detail view survived, where before it collapsed), sent a message from the composer; it delivered, claude replied, conversation continued in the same thread.
**Tier:** Product (shipping iOS app + server). Verified with real transport (API) + real gesture (simulator), per [[verify-real-seam-not-mocks]] and [[verify-ui-by-tapping]].

### 2026-06-29 — Bug fixed: sends dropped on leaving view / backgrounding

**Root cause:** The composer dispatched the send as a view-owned `Task { await store.sendWithAttachments(...) }`. Two exposures: (1) the message delivery is only as durable as whatever is keeping the task alive while the network round-trip is in flight — and my own auto-resume change widened that window to ~6s for closed sessions; (2) no background-task assertion, so pressing Home suspended the in-flight POST and the message was lost.
**Fix (iOS `SessionStore` + `SessionDetailView`):**
- `dispatchSend` owns the send in the store (retained in `inflightSends`), so leaving the view / popping the nav stack can't drop it.
- The UIKit background-task assertion (`beginBackgroundTask`) is taken **synchronously** inside `dispatchSend` — BEFORE it returns — not inside the async Task. First attempt took it inside the Task and the message was still lost on Home (the assertion raced app suspension). Taking it synchronously fixed it.
- Removed the now-dead view-owned `sending` spinner state; the optimistic bubble + pending strip are the feedback.
**Verification (iOS simulator, real gesture):** With a closed session (so the send triggers a ~6s server resume = wide in-flight window): (a) send then immediately tap Back → delivered, session resumed, Claude replied; (b) send then immediately press Home → delivered, session resumed, Claude replied. Before the synchronous-assertion fix, (b) lost the message — a clean before/after.
**Note:** The iOS Simulator doesn't suspend like a device, so backgrounding fidelity is limited — but the message-loss reproduced and the synchronous assertion resolved it, which matches device semantics.

### 2026-06-29 — Bug fixed: notification tap lands on "No session selected"

**Root cause:** Tapping a push notification cold-launches (or wakes) the app and routes via `store.requestSelection(sid)` → RootView `.onChange(of: requestedSelection)`. On cold launch the tap is processed during startup, BEFORE that `.onChange` observer attaches — and `.onChange` doesn't fire for a pre-existing value — so the selection was dropped. The session list also isn't loaded yet, and a since-closed session is never in the live list at all.
**Fix (iOS `SessionStore`, `RootView`, `PushManager`):**
- `openFromNotification(sid)` (replaces the bare `requestSelection` for taps): sets the selection AND actively reconnects — `refresh()` to load the live list, and if the session has closed, pulls it from the resumable list into a `deepLinkSession` fallback so its transcript still opens (sending revives it via the existing auto-resume).
- RootView `.onChange(of: requestedSelection, initial: true)` so a selection set before the observer attaches (cold launch) is honored on first appearance.
- New `DetailLoading` ("Opening session…") shown while the deep-linked session resolves, instead of the "No session selected" empty state.
- Refresh on `scenePhase == .active` so the app reconnects the instant it foregrounds (iOS can't hold the connection while suspended).
**Verification:** Builds clean, no regression (app reconnects + list loads). The actual notification-tap hop could NOT be synthetically automated — iOS 26's lock-screen/banner activation doesn't respond to FlowDeck's injected taps (verified: app-icon taps work, notification-cell taps register but don't launch). Left to user/device verification. The resolution machinery (refresh + resumable + selection→detail) is the same verified E2E in the earlier fixes; `onChange(initial:true)` is the standard fix for the missed-pre-set-selection race.

### 2026-06-29 — Bug fixed: momentary duplicate message on pickup

**Root cause:** An idle send shows an optimistic `showSent` bubble (rendered by a separate `ForEach(pending…)`) until the store's `reconcilePending` removes it. When the real user turn streams in, it's appended to `messages` and reconcile clears the placeholder — but for a render beat the placeholder `ForEach` and the real-message `ForEach` both show the same text ("momentary duplicate") before the pendingSends mutation propagates.
**Fix (iOS `SessionDetailView`):** Render the optimistic bubbles from a computed `unmatchedSentBubbles` that filters `pending` against the live `messages` (same normalize/prefix match as the store). The placeholder now drops out in the SAME render pass the real turn appears — no overlap.
**Verification:** Built + sent a message to a live idle session in the simulator; the user turn rendered exactly once and the agent reply followed cleanly. (The pre-fix transient is sub-500ms so not frame-captured, but the fix removes the render window by construction.)

### 2026-06-29 — Bug fixed: notification tap crashes the app on launch

**Symptom progression:** notification tap → "No session selected" (pre-fix) → black screen (after `onChange(initial:true)`) → crash "opens then quits" (after `.task` change) → working (final).
**Root cause (from the device/sim crash log, NOT guessed):** `LFG-2026-06-29-*.ips` showed `EXC_CRASH (SIGABRT)`, an uncaught `NSException` thrown inside `AppDelegate.userNotificationCenter(_:didReceive:)` during UIKit's `_updateSnapshotAndStateRestoration`. The handler set the `NavigationSplitView` selection **synchronously inside UIKit's launch/CATransaction commit**; mutating nav state mid-transaction makes UIKit throw during its state-restoration snapshot. `initial:true` blanked the screen (state mutation during view update); `.task` didn't help because `onChange` still fired synchronously within the transaction.
**Fix:** `SessionStore.openFromNotification` defers setting `requestedSelection` (and the downstream selection/navigation) onto the next main-actor runloop turn via `Task { @MainActor in }`, so it runs AFTER UIKit's launch transaction — navigating like a normal in-app tap.
**Verification:** User confirmed on-device: tap now shows a loading state, then loads the session + messages. Cleanly.
**Lesson:** I shipped the notification deep-link without being able to verify the actual tap (iOS 26 simulator won't let injected taps activate notifications), which cost two on-device iterations (black → crash). For a code path that can't be simulator-verified, pull the crash log on the FIRST device failure (ground truth) instead of reasoning forward — the `.ips` in `~/Library/Logs/DiagnosticReports/` (sim) named the exact throw site immediately. Also: any state mutation triggered from `didReceive` / app-launch callbacks must be deferred off the UIKit transaction.
