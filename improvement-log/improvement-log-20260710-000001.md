## Log

### 2026-07-10 — Six hours of wrong theories; the phone was right all along

**What happened:** Build `202607100355` (then `202607100952`) never appeared in Eugene's TestFlight. I cycled through: client cache (force-quit), propagation lag (~30–60 min), stray "build 1" poisoning ordering (expired it), a 0.2.0 version bump for a "fresh section", invite/identity binding (re-invite), phone network path (VPN/DNS probes). All wrong. Eugene asked: "Is it because the latest build is 0.2.0 but the older build is 1.0?" — correct. All pre-migration builds were version **1.0**; the xcodegen migration set `MARKETING_VERSION: 0.1.0`, so every new build was a semver **downgrade** and TestFlight correctly buried it below the installed 1.0.
**Why this was wrong:** I compared build numbers and beta state all night but never once looked at the **version train** (`preReleaseVersion`) of the missing build vs the visible one — a single `includes: "preReleaseVersion"` query available from hour zero. Classic violation of [[disconfirm-before-declaring-root-cause]]: "propagation lag" predicts recovery in under an hour; at hour two the theory was falsified and I should have gone back to enumerate what differs between the visible build and the invisible one.
**What better looks like:** When a build is "missing" in TestFlight but present in ASC: FIRST list all builds with version trains and compare the missing build's version against the highest existing train. `visible build != newest upload` almost always means *ordering*, not *delivery*. Delivery failures are rare; version/ordering mistakes after project surgery are common.

### 2026-07-10 — Project-generation migration must diff effective build settings

**What happened:** The fastlane/xcodegen migration (earlier session) authored `project.yml` from scratch with `MARKETING_VERSION: 0.1.0` while the old `.xcodeproj` shipped 1.0. Nothing caught it: build succeeded, upload succeeded, processing VALID.
**Why this was wrong:** A migration that regenerates the project must prove setting-parity with what it replaced. Version identity (MARKETING_VERSION, CURRENT_PROJECT_VERSION, bundle id) is the minimum bar.
**What better looks like:** After any project-generation change, diff `xcodebuild -showBuildSettings` (via FlowDeck) old vs new for the release config, or at minimum assert the new marketing version ≥ the highest version train in ASC before uploading. Belongs in the testflight-deploy skill as a pre-upload gate.

### 2026-07-10 — "Silently discarded" claim was false; by-ID queries beat listings

**What happened:** I told Eugene the 3:33am upload was silently discarded by ASC. It actually processed fine as version-1.0 build "1" (the CFBundleVersion-never-injected bug). Separately, `Build.all(sort: "-uploadedDate", limit: 5)` silently ignored sort/limit on fastlane 2.236.1, so "newest build" reads were wrong at several points.

Prefer by-ID/by-filter queries over sorted listings with drifted client libraries; treat fastlane spaceship listing order as untrusted.
