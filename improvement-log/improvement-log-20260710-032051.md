### 2026-07-10 — Phase 1 declared done with the other host's fix unpushed

**What happened:** `.claude/feature/phase1-connectivity-core.md` marks SC1–SC8 all ✅ with live
evidence. But `git rev-list --left-right --count origin/main...HEAD` → `0 14`. Every Phase-1 commit,
plus `421e07d` ("journal-pump: serialized self-scheduling loops — **fixes Air wedge**"), is
local-only on the Pro. The Air has been wedged since, and could not have pulled the fix if it tried.

**Why this was wrong:** Phase 1 is a *multi-host* connectivity feature. It was verified end-to-end
on exactly one host. The verification gates (SC1–SC8) all ran against a local server. "Green on the
machine I'm sitting at" is not verification for a feature whose entire subject is the second machine.
The project CLAUDE.md warns about *running code lagging source*; this is the sibling class — **source
lagging on the other host, because it was never pushed.** Same root: nobody checked the other end.

**What better looks like:** For any change that spans hosts, the definition of done includes
`git push` **and** a probe of the remote host's actual behavior (here: `nc -v <air> 8766` connects and
`/api/ping` returns 200 — that endpoint exists only in Phase 1, so a 200 doubles as a version
fingerprint). Add a "deployed to all hosts" line to the feature-doc template alongside the SC gates.

### 2026-07-10 — Didn't check git history for prior art before probing

**What happened:** Spent several probe rounds establishing the wedge signature (RST on 22/80 vs
timeout on 8766, 502-after-8s through `tailscale serve`). Then found `421e07d`'s commit message
already stated it verbatim: *"TCP accepted via kernel backlog, HTTP never answered — the exact
signature."*

**Why this was only half-wrong:** The independent probe was genuinely valuable — it disconfirmed the
macOS-firewall hypothesis (the loopback 502 rules it out), per memory `disconfirm-before-declaring-root-cause`.
Ground truth first was correct. But a 5-second `git log --oneline --grep=Air` at the *start* would
have named the hypothesis to test, making the probing targeted instead of exploratory.

**What better looks like:** For a recurring symptom on a known component, run
`git log --grep=<component-or-symptom>` before designing probes. Use prior art to *generate* the
hypothesis; still run the probe to *disconfirm* it. Cheap, and it doesn't cost independence.

### 2026-07-10 — The Air has no remote shell

**What happened:** `ssh eugenechan@100.75.162.40` → Connection refused (Remote Login off).
`tailscale ssh` → falls back to plain ssh, also refused. Could not read the Air's `git log`, its
running process, or its `.env`. The entire diagnosis had to be done through black-box network probes.

**Why this matters:** The second host in a multi-host system is undeployable and undebuggable from
the primary. Every future Air issue costs 10× to diagnose. This is infrastructure debt, not a
one-off inconvenience.

**What better looks like:** Enable Remote Login on the Air (System Settings → General → Sharing) or
`tailscale set --ssh`. Then a one-line `lfg deploy <host>` script that pulls and restarts the
supervisor. Should have been part of the multi-host work, not discovered after it.

### 2026-07-10 — `serve-forever.sh` can't see a wedged child

**What happened:** `scripts/serve-forever.sh` restarts the child only on process **exit**. The Air's
server is bound, listening, and never answering — a live process with a drowned event loop. The
supervisor sees a healthy child forever. The Air's occasional reconnects are Bun segfaulting (the
documented reason the supervisor exists), not the supervisor doing its job.

**Why this matters:** The supervisor's existence created false confidence that the Air self-heals.
It only heals crashes, not hangs — and hangs are the failure mode this codebase actually has
(single-threaded Bun event loop; see the project CLAUDE.md architecture hazards).

**What better looks like:** Liveness ≠ process-exists. Add a loopback health probe to the
supervisor: `curl -m 2 127.0.0.1:$PORT/api/ping` every 30s, `kill -9` the child after N consecutive
failures. Backpressure fixes *this* pileup; a watchdog covers the next one. Generally: any
supervisor for a single-event-loop process must probe the loop, not the PID.
